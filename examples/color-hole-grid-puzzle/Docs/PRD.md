# PRD: Color-Hole Grid Puzzle

## Product Vision

짧은 이동 시간이나 휴식 시간에 한 손으로 공간 퍼즐을 즐기고 싶은 플레이어가, 복잡한 설명 없이 블록의 모양과 색을 읽고 통로를 계획해 1~3분 안에 작은 해결의 쾌감을 얻는 모바일 게임을 만든다. 여러 칸을 차지하는 수집 블록을 직접 끌어 같은 색 승객을 채우는 상호작용은 즉시 이해되지만, 부분 충전된 블록이 계속 공간을 차지하기 때문에 다음 경로를 생각하게 만드는 것이 핵심 가치다.

이 제품은 특정 상용 게임의 재현물이 아니다. 장르에서 발견한 “형태가 있는 블록을 격자에서 움직여 색을 맞춘다”는 추상적인 아이디어만 출발점으로 삼고, 최종 제목, 세계관, 캐릭터, 시각 스타일, 규칙 조합, 레벨과 밸런스는 독자적으로 만든다.

## Core Value Sentence

플레이어는 복잡해 보이던 보드에서 블록 하나의 경로를 찾아 승객을 차례로 채우고 공간이 열리는 순간, 자신의 계획이 눈앞에서 맞아떨어졌다는 짧고 선명한 만족을 느낀다.

## Target User

초기 사용자는 세로 화면의 캐주얼 퍼즐을 한 세션에 3~10분 정도 즐기며, 긴 튜토리얼이나 빠른 반사 신경보다 관찰과 경로 계획을 선호하는 모바일 플레이어다. first proof는 퍼즐 장르에 익숙하지 않은 사람도 첫 드래그 후 30초 안에 목표를 설명할 수 있는지를 우선 확인한다.

## First Playable Proof

첫 proof에는 한 화면의 작은 격자, 고정된 1×2 수집 블록 하나, 같은 색 승객 두 명과 최소한의 충돌 경계가 있다. 플레이어는 블록을 직교 방향으로 드래그하고, 이동 경로에서 승객을 한 명씩 수집해 부분 충전 상태를 확인하며, 두 슬롯이 모두 차면 블록이 제거되고 승리 화면에서 재시작할 수 있다.

다음 조건이 모두 재현되어야 proof로 인정한다. playable scene에서는 같은 입력이 프레임률과 드래그 속도에 관계없이 같은 상태를 만들고, 보드 밖으로 나가거나 대각선 모서리를 자르지 않아야 한다. 한 명만 수집한 블록은 보드에 남고, 두 번째 승객까지 수집한 뒤에만 논리적으로 제거된다. 다른 Hole, wall과 다른 색 승객 충돌은 별도의 순수 모델 fixture로 검증하고 playable scene의 범위를 키우지 않는다. 승리와 재시작은 최소 한 번의 실제 터치 세션과 자동 테스트에서 모두 확인한다.

## MVP Scope

MVP는 여러 개의 고정 폴리오미노 블록, 여러 색의 승객, 데이터 기반 레벨 로드, 부분 충전과 제거, 승리·실패, 재시작, 일시정지, 레벨별 선택적 타이머, 기본 피드백과 10~20개의 자체 제작 레벨까지 포함한다. 블록의 모양과 색을 색각 차이에 관계없이 읽을 수 있도록 패턴 또는 아이콘을 함께 사용하며, 논리 규칙과 View를 분리해 테스트 가능한 구조를 유지한다.

first proof가 검증되기 전에는 타이머와 특수 장애물을 추가하지 않는다. proof 이후의 MVP 후보도 `Docs/AcceptanceTests.md`에 검증 가능한 규칙이 추가된 경우에만 구현한다.

## Explicitly Out Of Scope

첫 공개 테스트 전에는 광고 SDK, 인앱 결제, 코인 경제, 계정, 리더보드, 원격 서버, 라이브 운영, 자동 레벨 생성, 복잡한 solver, 대규모 에셋 팩을 만들지 않는다. 외부 게임의 레벨을 전사하거나 캡처를 배경으로 사용하는 일, 원작과 혼동될 이름·캐릭터·UI·사운드·스토어 문구를 사용하는 일은 모든 단계에서 범위 밖이다.

## Core Loop

```text
보드 읽기 -> 블록 선택 -> 안전한 경로로 이동 -> 같은 색 승객 수집
-> 블록 완성 및 제거 -> 새 통로 확인 -> 남은 블록 해결 -> 결과 확인 -> 다음 레벨
```

## UX Quality Bar

주요 블록과 목표는 세로형 휴대전화 화면에서 즉시 식별되어야 하고, 손가락 아래에서도 선택 상태와 이동 가능 방향을 알 수 있어야 한다. 유효하지 않은 이동은 블록이 튀거나 순간 이동하는 대신 마지막 유효 칸에 안정적으로 멈추고, 수집·부분 충전·완료는 서로 다른 시각 및 촉각 피드백으로 구분한다. 색만으로 규칙을 전달하지 않고 모양, 패턴, 아이콘 중 하나를 중복 신호로 제공한다.

애니메이션은 입력 반응을 지연시키지 않으며 논리 결과를 가리지 않는다. 작은 화면에서 최소 터치 영역과 safe area를 확인하고, 타이머를 도입할 때는 일시정지와 앱 background 전환 중 시간이 잘못 감소하지 않도록 한다. 사운드와 진동은 각각 끌 수 있어야 하지만 first proof에서는 설정 UI보다 명확한 기본 피드백을 우선한다.

## Data And Privacy

first proof와 MVP는 계정이나 서버 없이 로컬 레벨 데이터와 로컬 진행 상태만 사용한다. 개인정보와 광고 식별자를 수집하지 않고, 분석 SDK는 실제 검증 질문과 보존 정책이 정해진 이후에만 검토한다. crash 도구나 TestFlight를 사용할 때도 공개 문서에는 실제 tester 정보와 계정값을 남기지 않는다.

## Validation Path

첫 검증은 Unity EditMode의 규칙 테스트, PlayMode의 드래그 smoke test, 세로형 시뮬레이터와 실제 기기 확인 순서로 진행한다. 이후 3~5명의 소규모 테스트에서 설명 없이 첫 목표를 이해하는지, 첫 레벨을 완료하는지, 실패 원인을 말할 수 있는지 관찰한다. 관찰 결과는 개인 식별정보 없이 iteration 문서에 요약하고, 기능 수보다 핵심 이동의 이해도와 조작 안정성을 먼저 개선한다.

## Release Path

첫 공유 단위는 commit hash가 연결된 Editor 또는 기기 playable build다. core loop와 실제 터치가 안정된 뒤 iOS TestFlight 또는 동등한 내부 테스트로 확장한다. `<UNITY_VERSION>`, `<BUNDLE_ID>`, version과 build number는 대상 프로젝트에서 관리하고, kit 예시에는 실제 계정값을 기록하지 않는다.

## Learning Questions And Stop Conditions

가장 먼저 확인할 질문은 플레이어가 별도 설명 없이 어떤 블록으로 어떤 승객을 모아야 하는지 이해하는가, 드래그 결과를 예측할 수 있는가, 부분 충전이 다음 경로 선택에 의미 있는 제약을 만드는가이다. 세 명 이상의 연속 관찰에서 첫 행동을 이해하지 못하거나, 빠른 드래그의 결과가 반복 실행마다 달라지거나, 핵심 규칙을 설명하기 위해 여러 화면의 텍스트가 필요하다면 기능 추가를 중단하고 입력 피드백과 규칙 표현을 다시 설계한다.

첫 proof 완료 전에는 신규 장애물이나 메타 기능 PR을 받지 않는다. MVP에서도 새 기능이 core loop의 선택을 늘리지 않고 단순히 원작과 비슷하게 보이기 위한 것이라면 제외한다.

## MVP Done Criteria

자체 제작 레벨 세트가 스키마 검증을 통과하고, 필수 규칙 자동 테스트와 실제 기기 smoke test가 통과하며, 신규 플레이어가 도움 없이 첫 레벨을 완료할 수 있어야 한다. crash 없이 한 세션을 마치고 재시작과 앱 복귀가 정상 동작해야 하며, 공유 빌드의 commit hash와 검증 결과가 handoff에 기록되어야 한다. 스토어에서 원작과 혼동될 표현물이나 권리 확인이 안 된 에셋이 없어야 완료다.
