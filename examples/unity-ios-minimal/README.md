# Unity iOS Minimal Example

이 폴더는 실제 Unity 프로젝트를 포함하지 않습니다. 대신 에이전트가 새 Unity iOS 프로젝트를 만들 때 유지해야 할 최소 구조를 설명합니다. 공개 kit repo에 큰 binary asset이나 개인 signing 파일을 넣지 않기 위해 예시는 문서 중심으로 둡니다.

## First Unity iOS Proof

첫 proof는 한 화면, 한 조작, 한 결과로 좁힙니다. 게임이라면 캐릭터 하나, 적 하나, 보상 하나, 성장 한 번처럼 사용자가 가치를 느끼는 가장 작은 루프를 닫습니다. 앱이라면 입력 하나, 결과 하나, 저장 또는 공유 하나로 충분합니다.

## Growing a Game Mechanic in Slices

게임 메커니즘(예: 드랍 → 조합 → 성장)을 추가할 때는 한 번에 다 만들지 말고 vertical slice로 끊습니다.

- **순수 모델과 UI를 분리한다.** 드랍/머지/장착 같은 규칙은 UnityEngine에 의존하지 않는 순수 C# 클래스로 두면 결정적이고 테스트 가능합니다. UI(슬롯·아이콘)는 그 모델을 읽어 그리기만 합니다.
- **slice마다 컴파일 + 시뮬레이터 검증.** Slice 1(드랍 → 슬롯 표시) → Slice 2(같은 것 머지 → 상위 tier) → Slice 3(색 패널 → 실제 아이콘)처럼, 각 단계를 배치 컴파일과 sim 스크린샷으로 확인하고 넘어갑니다.
- **기존 시스템 위에 점진적으로.** 새 성장(무기)을 기존 성장(강화)과 `max()`로 병행하면, 코어를 한 번에 갈아엎지 않고도 새 재미를 얹을 수 있습니다.
- **에셋은 반복 생성.** 아이콘·스프라이트는 PixelLab 같은 도구로 만들고, 어긋나면(예: 활이 막대처럼 나옴) 프롬프트를 고쳐 그 하나만 재생성합니다. 애니·스프라이트 생성의 구체적 파이프라인과 함정은 `03-ios-unity-setup/PixelArtAnimationPipeline.md`를 참고하세요.

## Reworking the Core Loop in Slices (static → moving)

작은 피처만 slice로 끊는 게 아니라 **코어 루프 자체를 크게 바꿀 때도** 같은 방식이 안전합니다. 정적 1:1 전투("탭해서 공격")를 던전을 이동하며 싸우는 자동 크롤러로 옮긴 경험:

- **상태 머신으로 나눈다.** `Advancing`(배경 스크롤 + walk 애니) ↔ `Fighting`(멈춰서 자동 공격)처럼 상태를 명시하면, 이동과 전투를 한 덩어리로 섞지 않고 각 상태를 따로 검증할 수 있습니다.
- **기존 규칙을 재사용한다.** 데미지·HP·처치·드랍/성장 규칙(순수 모델)은 그대로 두고 위에 이동·조우·자동 진행만 얹습니다. 코어를 새로 쓰지 않아 회귀가 적습니다.
- **수동 입력을 자동으로 대체.** "탭해서 공격"을 타이머 기반 자동 전투로 바꾸면, 중복된 수동 버튼(CTA)은 숨깁니다. idle 게임에 맞고 조작이 단순해집니다.
- **slice로 옮긴다.** Slice 1(배경 세로 스크롤 + walk 애니, 적 없음) → Slice 2(조우 → 자동 전투 → 처치 → 재개) → Slice 3(연속 몬스터·보스·성장 결합). 각 단계를 컴파일 + sim으로 확인하고 넘어갑니다.
- **자기 점검(self-check)은 sim 전용으로 취급.** 코어를 크게 바꾸면 UI 문자열에 묶인 검증이 깨질 수 있습니다. 그런 검증은 시뮬레이션 전용으로 두고 프로덕션과 분리하면, 재설계가 검증에 발목 잡히지 않습니다.

## Difficulty & Boss Curve by Reusing Assets

난이도 곡선과 보스는 **이미 만든 에셋을 조건 분기로 재사용**하면 적은 비용으로 붙습니다.

- **진행도에 따라 스프라이트/배경 교체.** 예: 일반 몹 → 강한 몹(깊이 3+) → 보스(깊이 5+), 배경도 풀밭 → 화산 아레나. 새 로직 없이 `SpriteForDepth(depth)` 같은 분기 하나로 시각이 확 달라집니다.
- **보스는 값 몇 개로 "진짜 싸움"처럼.** 5의 배수 깊이만 보스로 잡아 HP를 배수로 올리고, 라벨 색을 바꿔(예: 초록 → 빨강) 신호를 줍니다. 전용 보스 스프라이트가 없어도 기존 강한 몹을 크게·틴트해 대신할 수 있습니다.
- **스크롤 배경은 타일 두 장을 같은 스프라이트로.** 세로 스크롤을 두 장 번갈아 그릴 때 깊이가 바뀌어 배경 스프라이트를 교체하면 **두 장 모두** 바꿔야 합니다. 한 장만 바꾸면 스크롤 도중 두 배경이 번갈아 깜빡입니다(실제로 겪은 버그).

## Rhythm & Stakes (테스터 피드백에서 나온 세 가지 수정)

자동 진행 게임을 실제 사람에게 보여주면 "돌아는 가는데 재미가 안 붙는" 지점이 드러납니다. 실기 테스트 피드백에서 나온 세 가지 구조 수정:

- **맺고 끊음(스테이지 + 정비).** 무한 자동 진행은 리듬이 없습니다. "N마리 처치 = 스테이지 클리어 → 자동 진행 멈춤 → 정비 화면 → Next" 구조로 끊으면, 전투(자동)와 성장(수동)의 리듬이 생깁니다. 구현은 순수 모델에 `StageKills/StageTarget/IsStageClear`, 컨트롤러에 일시정지 가드 하나면 됩니다(컷신 일시정지와 같은 패턴).
- **성장은 유저의 손으로.** 드랍이 자동으로 합쳐지면 성장이 "배경 규칙"이 되어 체감이 없습니다. 자동 머지를 폐지하고 정비 화면에서 유저가 같은 아이템을 탭해 직접 합치게 하면, 같은 로직이 능동적 선택이 됩니다. 성장의 재미는 로직이 아니라 **누가 버튼을 누르느냐**에서 나옵니다.
- **전투에 판돈(양방향).** 적이 반격하지 않으면 성장할 이유가 없습니다. 플레이어 HP + 적 반격 + 패배 시 후퇴(성장은 유지)로 "벽에 막힘 → 성장해서 돌파" 루프를 만듭니다. 처치 시 소량 회복을 넣으면 "빨리 죽이면 유지, 못 크면 소모"가 자연스럽게 성립합니다.
- **보상은 명시적으로, 소비는 선택으로.** 킬이 주는 것(XP·드랍)을 숫자와 배너로 화면에 찍고, 그 자원을 쓰는 곳(강화/회복/진행)을 나란히 놓으면 같은 로직이 "경제"로 읽힙니다. 테스터 피드백에서 반복된 요구는 기능 추가가 아니라 **이미 있는 보상·성장의 가시화**였습니다 — "적 잡을 때마다 경험치·드랍을 보여달라", "스테이지 끝에 선택지를 달라".
- **보상 배너의 실전 디테일 세 가지.** ① 게임 화면 위에 뜨는 텍스트는 **어두운 반투명 백드롭 패널 위에** — 밝은 구역(용암 등)에선 맨 텍스트가 안 읽힙니다. ② 가변 길이 목록(드랍 여러 개)은 **줄을 나눠서**(XP 한 줄, 드랍 한 줄) — 한 줄로 이으면 화면 폭을 뚫고 잘립니다. ③ 표시 시간은 애니메이션 기준이 아니라 **읽기 기준**으로 — 두 줄이면 2초 이상. 그리고 이런 1~2초짜리 순간 UI는 고정 간격 스크린샷 sim으로는 못 잡습니다. 억지로 잡으려 말고, 그 코드 경로가 매 팩마다 무오류로 도는지(진행이 계속되는지)로 간접 검증하는 게 쌉니다.
- **소비처는 막힌 상태에서도 닿아야 합니다(딥런으로 발견한 데드락).** 자원(XP)을 벌 수는 있는데 쓰는 곳이 "스테이지 클리어 후 정비 화면"뿐이면, **벽에 막힌 플레이어는 영원히 소비처에 못 갑니다** — 자원만 쌓이고 성장은 동결된 채 같은 벽에 계속 부딪히는 데드락. autoplay 딥런에서 XP 405가 사장되고 데미지가 동결된 화면으로 잡혔습니다. 수정은 새 UI가 아니라 **기존 정비 화면을 패배 후에도 여는 것**("재정비 — 재도전" 제목만 교체): 패배마다 쌓인 자원을 성장으로 바꾸고 재도전하니 "벽→성장→돌파"가 기계적으로 성립합니다(같은 딥런에서 데미지 66→146 상승으로 검증). 경제를 설계했다면 "**모든 막힌 상태에서 소비처까지의 경로가 있는가**"를 딥런으로 확인하세요.
- **선택 축은 대칭으로 늘리면 싸다.** 무기에 "kind = 스타일"(강공/안전/흡수/균형)을 넣어
  장착이 선택이 되게 했다면, 방어구도 **같은 탭 패턴·같은 opt-in 규칙**으로 방어 스타일
  (반격 감소/선제 감소/패배 손실 감소)을 붙일 수 있다 — UI·인터랙션 코드는 재사용이라
  추가 비용이 거의 0인데 조합 공간은 곱으로 는다(공격 4 × 방어 3 = 12 빌드). 단 두 축의
  같은 항 감소(예: 반격 -25% × -15%)는 곱으로 쌓이므로 최대 생존 조합을 관찰 항목으로 남길 것.
- **소비 버튼에는 비용을 쓰고, 못 사면 잠급니다.** 강화 같은 소비 액션이 무료·무제한이면 경제가 통째로 무너집니다(실기 테스터가 바로 발견한 실수 — 비용 도입 전 빌드가 나감). 버튼 라벨에 비용을 명시("+1 · 30 XP")하고, 잔액 부족이면 (a) 순수 모델이 거부하고 (b) UI는 흐림 처리 — 게이트는 모델에, 표시는 UI에 두면 우회 경로가 없습니다.
- **자동 성장이 판돈을 죽입니다(직접 겪은 함정).** 성장(강화)이 자동으로 돌면 데미지가 난이도 곡선을 그대로 추격해 모든 적이 원샷—반격 기회 자체가 없어 위 판돈 루프가 무력화됩니다. 반격/HP를 다 만들어도 "안 죽는" 원인이 밸런스 값이 아니라 **자동 성장 흐름**일 수 있습니다. 성장을 유저 액션(정비 화면의 강화 버튼)으로 옮기면 성장 속도가 유저 페이스로 묶여 벽·패배·돌파가 실제로 생깁니다. 검증은 autoplay sim(정비·컷신을 자동 탭하는 sim 전용 환경변수 훅)으로 깊은 구간까지 돌려 패배 연출이 화면에 찍히는지 확인했습니다.

## Cheap Enemy Variety (팩 + 특성 틴트)

전용 에셋 없이 전투 다양성을 만드는 두 가지:

- **조우 팩(1~N마리).** 조우마다 적 수를 결정적으로 바꾸고(예: `1 + kills % 3`), 처치하면 다음 멤버가 등장. 장식용이던 옆 몬스터를 "대기 중인 팩 멤버" 표시로 재활용하면 새 에셋이 필요 없습니다.
- **특성은 계수 + 틴트로.** 같은 스프라이트에 색만 입혀(강타형=붉게·공격 1.5배/HP 0.8배, 방어형=푸르게·공격 0.7배/HP 1.6배) 종류를 구분하면, 데미지 공식의 항 하나씩만 건드리는 적 특성이 값싸게 생깁니다. 종류 배정도 `kills % 5` 같은 결정적 규칙이면 테스트 가능합니다.
- **보스는 스테이지의 클라이맥스에.** 보스를 "깊이 5의 배수"처럼 진행도에 걸면 팩/스테이지와 어긋나 어중간한 타이밍에 나옵니다. "매 3번째 스테이지의 마지막 몹"처럼 **스테이지 경계에 정렬**하면 정비 직전의 클라이맥스가 됩니다.

## Cutscenes as Data (대본과 재생 기계의 분리)

역전재판식 텍스트 컷신(타이핑 연출 + 선택지)을 넣을 때, 대본을 코드에 흩뿌리지 말고
**struct 데이터**(Speakers/Texts/Choices/ChoiceResults 배열)로 분리하면:

- **대본 추가 = 코드 무수정.** 지역/상황별 분기도 `ScriptForStage(stage)` 같은 선택 함수
  하나로 끝난다(보스 컷신을 지역 3종으로 늘릴 때 재생 코드는 한 줄도 안 바뀌었다).
- **재생 기계는 한 번만 만든다.** 실제로 까다로운 건 대본이 아니라 기계다 — 타이핑 연출,
  선택지 대기 가드(선택 대기 중 화면 탭이 컷신을 닫아버리는 버그 — 실기에서 발견),
  1회성 재생(PlayerPrefs 플래그), 컷신 중 전투 일시정지. 이걸 데이터 주도로 한 번
  검증해두면 이후는 대본 작업만 남는다.
- **톤 리뷰를 코드 밖에서.** 대본 뱅크 문서(화자 성격표 + 문체 규칙 + 복붙 가능한 대본
  블록)를 두면 팀원·작가가 코드를 열지 않고 대사를 리뷰/추가할 수 있다. 문체 규칙엔
  "밈은 한 컷신에 1개" 같은 억제 규칙도 함께 — 밈 남발은 톤을 무너뜨린다.
