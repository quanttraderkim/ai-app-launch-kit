# Case Study: 실기 피드백 5건이 세 빌드가 되기까지

한 사이드 프로젝트(픽셀 자동 크롤러)의 실제 3일 기록. 테스터 피드백 5건이 어떻게
핫픽스 1건 + 대형 개편 1건 + 설계 수정 여러 건으로 갈라졌는지, 그리고 각 단계에서
어떤 판단을 했는지를 남긴다. 숫자·이름은 프로젝트마다 다르니 **판단의 순서**를 보라.

## 1. 피드백 트리아지 — 심각도가 아니라 "차단 여부"로 나눈다

테스터 피드백 5건: ① 버튼이 안 눌림 ② 적 공격 모션이 안 보임 ③ 컷신 대사 겹침
④ 타이틀 화면 요망 ⑤ 게임성 요망.

- ①은 **차단 버그**(핵심 인터랙션 불가) → 당일 핫픽스 대상.
- ②③은 연출 결함 → 같은 핫픽스에 동승.
- ④⑤는 방향 요구 → 다음 정규 빌드의 설계 작업으로.

## 2. 스크린샷 없이 근본 원인 잡기 — 프레임워크 규칙에서 역산

"버튼이 안 눌림"은 sim 스크린샷으로는 재현이 안 된다(보이는 건 정상). UGUI의
레이캐스트 규칙에서 역산: 버튼 위에 **형제(sibling)로 얹힌 Text가 탭을 삼킨다**
(`Text.raycastTarget` 기본값 true). 라벨 생성 헬퍼에서 전역으로 끄는 1줄이 수정의
전부였다. 교훈: 실기 전용 버그는 "무엇이 그려졌나"가 아니라 "무엇이 입력을 받나"를 본다.

## 3. 핫픽스 판단 — 예외는 투명하게

배포는 하루 2회 창(처리 쿼터)이 원칙. 하지만 차단 버그 + 테스터가 지금 플레이 중 +
배포 창 안이라는 3조건이 겹쳐 당일 3번째 빌드를 올렸다. 예외는 만들 수 있다 —
단 즉시 보고하고, 원칙은 유지한다.

## 4. 경제 개편 — 테스터가 원한 건 기능이 아니라 가시화

"조합을 내가 하게 해달라, 보상을 보여달라, 선택지를 달라" — 전부 이미 있는 시스템의
**가시화** 요구였다. 킬 보상을 HUD·배너로 찍고, 소비(강화/회복/진행)를 정비 화면의
3버튼으로 나란히 놓으니 같은 로직이 경제로 읽혔다. 이때 테스터가 곧바로 "강화가
무제한"을 발견 — 비용 없는 소비 버튼은 경제를 통째로 무너뜨린다(비용은 순수 모델이
게이트, UI는 흐림 표시).

## 5. 딥런이 설계 결함을 잡는다 — 두 번

autoplay 딥런(정비·컷신을 자동 탭하는 sim 전용 훅)으로 깊은 구간을 돌리다 두 가지
설계 결함이 화면에 찍혔다:

- **소비처 데드락**: 벽에 막히면 XP 소비처(스테이지 클리어 후 정비)에 영원히 못 간다
  — XP 405가 쌓인 채 데미지 동결. 수정은 새 UI가 아니라 기존 정비 화면을 패배 후에도
  여는 것("Regroup"). 같은 딥런에서 데미지 66→146 상승으로 돌파 검증.
- **기본값의 함정**: 무기 종류에 트레이드오프(도끼=반격 +25% 등)를 넣자, "자동 최강
  장착" 기본값이 신규 런에 도끼 페널티를 몰래 씌웠다. 특성은 **유저가 명시적으로
  선택했을 때만** 적용하도록(opt-in) — 시스템을 얹을 때 기본 경로가 어떻게 바뀌는지
  반드시 본다.

## 6. 배포 전 크래시 의혹 — 패닉 전에 재현

배포 6시간 전, 딥런 스크린샷에 게임 대신 홈 화면이 찍혔다(앱 소실 의심). 대응 순서:

1. 크래시 로그 검색(.ips/jetsam) → 없음.
2. **콘솔 로그를 문 재현 런**(simctl launch --console-pty + 생존 폴링) → 180초 완주,
   예외 없음.
3. 판정: 밤새 sim을 수십 회 돌린 호스트의 메모리 압박(일회성 환경 이슈). 배포 GO.

교훈: 이상 징후 1회 = 즉시 롤백이 아니라 **재현 시도**부터. 재현 런은 원래 잡으려던
다른 검증(보상 배너 스크린샷)까지 덤으로 끝내줬다.

## 요약: 판단 우선순위

1. 피드백은 차단 여부로 트리아지, 차단이면 당일(창 안에서) 해결.
2. 실기 전용 버그는 프레임워크 규칙에서 역산.
3. 요구의 절반은 기능이 아니라 가시화.
4. 경제·성장 시스템은 딥런으로 "막힌 상태"까지 검증.
5. 새 시스템의 기본 경로 변화를 의심.
6. 이상 징후는 재현부터, 배포 판단은 그 다음.
