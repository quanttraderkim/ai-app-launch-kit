# Pixel Art & Animation Pipeline

AI 픽셀아트 도구로 게임 에셋(캐릭터·아이콘·배경·프레임 애니)을 만들고 Unity에 넣는 실전 파이프라인. 예시는 PixelLab v2 API 기준이지만, 흐름과 함정은 다른 생성 도구에도 대체로 적용됩니다.

## 두 축: 정적 스프라이트 vs 애니메이션

| 용도 | 엔드포인트(예) | 크기 상한 | 참고 |
|------|----------------|-----------|------|
| 정적 스프라이트(캐릭터·아이콘·배경) | 이미지 생성 (pixflux) | ~400px | 프롬프트로 스타일 고정 |
| 프레임 애니 (walk·attack) | 텍스트 기반 애니 | 64px | reference_image 필수, 유료 tier |

**정적부터.** 캐릭터/적/아이콘을 한 장씩 생성해 스타일과 파이프라인을 먼저 검증합니다. 애니는 유료 구독이 필요할 수 있으니, 정적 에셋만으로 게임이 굴러가는 걸 확인한 뒤 얹습니다.

## 인증과 비동기 처리

- 토큰은 계정 페이지에서 발급받아 **환경변수로 주입**합니다. 스크립트나 repo에 하드코딩·커밋하지 마세요(→ `08-security`).
- 생성 엔드포인트는 대개 **비동기**입니다: 요청하면 job id를 돌려주고, 완료될 때까지 폴링해 결과를 받습니다. 스크립트에 폴링 루프를 넣어 두세요.

## 애니메이션의 함정 (실제로 막힌 지점)

- **크기 상한이 정적과 다릅니다.** 애니는 64px 상한이라 128px를 넣으면 거부(HTTP 422)됩니다.
- **reference_image는 출력과 같은 크기여야 합니다.** 128px 캐릭터를 reference로 넣고 64px 애니를 요청하면 텐서 크기 불일치로 실패(HTTP 500)합니다. 캐릭터를 먼저 64px로 리사이즈해 reference로 씁니다.
- **side view가 가장 자연스럽습니다.** 정면/탑다운이 필요하면 view·direction 파라미터를 따로 맞춰야 하고, 어긋나면 재생성이 필요합니다. 측면 캐릭터를 탑다운 던전에 그대로 쓰면 방향이 안 맞아 보이니 **실기에서 확인하고 다시 생성**하세요.
- **어긋난 에셋은 그 하나만 재생성.** 활이 막대처럼 나오는 등 특정 에셋만 이상하면 프롬프트를 고쳐 그 한 장만 다시 만듭니다. 전체를 다시 돌리지 않습니다.

## Unity 임포트

- 픽셀아트는 **filterMode = Point**로 임포트해야 흐려지지 않고 선명합니다. 스프라이트 `.meta`를 이미 잘 맞춰둔 에셋에서 복제하되 **guid는 새로 부여**합니다(중복 guid는 임포트 충돌).
- `Resources/Art/`에 두고 `Resources.Load<Sprite>("Art/<name>")`로 로드하면 코드빌드 UI에서 바로 씁니다.
- 넣은 뒤에는 **시뮬레이터 시각 확인**(→ `04-app-store-connect-testflight/VisualVerification.md`)으로 크기·정렬·색을 눈으로 검증합니다.

## 워크플로 요약

1. 정적 스프라이트 생성 → 다운로드 → `Resources/Art/` → meta(Point, 새 guid)
2. 코드에서 로드 → sim 스크린샷으로 확인
3. (필요 시) 캐릭터를 64px로 리사이즈해 reference로 애니 생성 → 프레임 분리 → 같은 방식으로 임포트
4. 애니 상태 전환(idle / walk / attack)은 코드에서 프레임 배열을 시간으로 순환

## 비용·범위 감각

- 애니·대량 생성은 크레딧을 씁니다. 먼저 최소 세트(캐릭터 1 + 적 1 + 아이콘 몇 개)로 루프를 닫고, 필요할 때 늘립니다.
- 정면 재생성·idle 추가 같은 후속 작업은 **실기 피드백을 본 뒤** 우선순위를 정하면 낭비가 적습니다. 시뮬레이터 정지 스크린샷으로는 애니 움직임이 안 보이니, 애니 확인은 `xcrun simctl io booted recordVideo`나 실기로 합니다.
