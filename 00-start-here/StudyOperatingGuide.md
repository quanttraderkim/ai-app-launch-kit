# Study Operating Guide

이 스터디의 목표는 2개월 동안 앱 하나를 “말로만 기획”하는 것이 아니라 실제 사용자가 만질 수 있는 상태까지 가져가는 것입니다. 모든 멤버는 작은 앱을 선택하고, 매주 동작하는 증거를 남기며, 마지막에는 TestFlight, 웹 배포, Android internal testing, 또는 오픈소스 릴리즈 중 하나의 공개 가능한 결과를 만듭니다.

## Weekly Cadence

매주 모임은 짧은 데모, 막힌 점 공유, 다음 주 proof 정의로 운영합니다. 발표는 멋진 슬라이드보다 실제 화면, repo 링크, build 링크, 실패 로그가 더 중요합니다. “무엇을 했다”보다 “무엇을 배웠고 다음 행동이 무엇인지”를 말합니다.

## 8-Week Milestones

1주차에는 아이디어, PRD, 환경 세팅, repo 생성까지 끝냅니다. 2주차에는 첫 proof를 만듭니다. 3주차에는 핵심 가치를 강화합니다. 4주차에는 사용자에게 보여줄 최소 흐름을 닫습니다. 5주차에는 TestFlight나 동등한 배포 lane을 확보합니다. 6주차에는 실제 피드백을 받습니다. 7주차에는 품질, 접근성, 크래시, 메타데이터를 정리합니다. 8주차에는 제출 후보를 고정하고 회고를 공유합니다.

## Demo Rule

데모는 항상 현재 repo commit 기준이어야 합니다. 로컬에서만 되는 임시 상태, 커밋되지 않은 빌드, 재현 불가능한 수동 조작은 데모의 보조 자료로만 취급합니다.

## Good Project Size

좋은 8주 프로젝트는 작고 선명합니다. “모든 사람을 위한 완성형 서비스”보다 “한 명의 사용자가 한 가지 문제를 해결하는 작은 앱”이 좋습니다. 첫 proof 전에는 로그인, 결제, 복잡한 서버, 대형 디자인 시스템, 과한 설정 화면을 피합니다.
